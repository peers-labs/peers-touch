# 全局 AI 架构与实施治理指引

## 1. 文档定位

### 1.1 治理目标
- 在 AI 参与编码场景下，统一架构决策、实现规范与交付质量门槛
- 将“功能完成”升级为“功能完成 + 工程一致性 + 可验证质量”
- 为规格、实现、评审、复盘提供同一套可执行基线

### 1.2 适用范围
- 适用于本仓库内全部新增功能、重要变更与跨模块改造任务
- 覆盖阶段：需求澄清、技术方案、编码实现、测试验证、上线验收、复盘演进
- 覆盖角色：架构师、产品工程负责人、开发、测试、评审者、AI 协作操作者

### 1.3 术语定义
| 术语 | 定义 |
| --- | --- |
| 治理主文档 | 统一约束 AI 编码与工程交付的一份基线文档 |
| 双视图 | 架构师视角与产品工程视角的并行说明结构 |
| 最小闭环 | 一个功能从立项到合并的最小必备动作和产物集合 |
| 硬性约束 | 必须满足，否则阻断合并的规则 |
| 可变项 | 在不破坏基线前提下允许按场景调整的项 |

### 1.4 角色职责
| 角色 | 核心职责 | 关键产物 |
| --- | --- | --- |
| 架构师 | 定义边界与硬性约束，裁决跨模块分歧 | 架构决策、约束清单 |
| 产品工程负责人 | 平衡业务交付与工程可维护性 | 实施计划、验收口径 |
| 开发/AI 协作操作者 | 依据规范实现并补齐验证证据 | 代码、测试、变更说明 |
| 测试/质量角色 | 验证行为、风险与回归影响 | 测试记录、质量结论 |
| 评审者 | 按治理清单进行合并前审查 | 评审意见、阻断结论 |

## 2. 双视图结构

### 2.1 架构师视角
#### 2.1.1 关注点
- 系统分层与边界稳定性
- 核心依赖和数据流正确性
- 技术债与演进风险可控

#### 2.1.2 决策输入
- 规格影响面
- 现有架构约束
- 历史一致性与兼容性要求

#### 2.1.3 决策输出
- 决策层级标注（架构层/模块层/实现层）
- 硬性约束列表
- 可变实现范围与禁用做法

#### 2.1.4 审查要点
- 是否跨越既定边界
- 是否引入未评估依赖
- 是否具备回滚与降级路径

### 2.2 产品工程视角
#### 2.2.1 关注点
- 功能目标可交付
- 开发联调与验收效率
- 可观测性与线上可运维性

#### 2.2.2 实施输入
- 需求与验收标准
- 接口契约和数据模型
- 风险与排期约束

#### 2.2.3 实施输出
- 任务拆解与里程碑
- 测试策略与验收证据
- 发布与回滚方案

#### 2.2.4 审查要点
- 功能路径是否可验证
- 异常路径是否有明确定义
- 交付物是否满足最小闭环

## 3. 功能实施最小闭环模板

### 3.1 模板使用规则
- 每个功能任务必须在开始实现前填写本模板
- 模板字段不得留空；若不适用须写明原因
- 合并前必须附上模板实例链接或内容摘要

### 3.2 模板正文
| 区块 | 必填项 |
| --- | --- |
| 基本信息 | 功能名称、需求来源、负责人、计划窗口 |
| 目标与边界 | 业务目标、非目标、影响模块、依赖前置 |
| 双视图对齐 | 架构师约束、产品工程实施口径 |
| 实施方案 | 接口契约、数据结构、异常语义、状态流转 |
| 验证方案 | 单元测试、集成测试、手工验证、监控点 |
| 发布与回滚 | 发布步骤、开关策略、回滚触发条件 |
| 交付证据 | PR 链接、测试结果、验收结论、已知风险 |

### 3.3 模板实例（最小可执行）
```md
#### 功能名称
统一治理文档落地

#### 目标与边界
- 目标：建立 AI 治理主文档基础结构
- 非目标：不在本次定义完整门禁与度量规则
- 影响模块：docs/architecture、.trae/specs

#### 双视图对齐
- 架构师约束：文档须提供分层边界承接位
- 产品工程口径：文档须可直接用于任务执行与验收

#### 实施方案
- 新增治理主文档骨架与章节占位
- 增加双视图章节（架构师/产品工程）
- 提供最小闭环模板与必填字段

#### 验证方案
- 文档结构完整性检查
- 任务状态与模板字段核对

#### 发布与回滚
- 发布：合并后作为默认治理入口
- 回滚：删除新增文档并恢复任务状态

#### 交付证据
- PR: 待补充
- 验证记录: 待补充
```

### 3.4 示例闭环（从规格到合并）
#### 3.4.1 目标与范围
- 示例功能：建立并落地“全局 AI 架构与实施治理指引”
- 规格入口：`.trae/specs/establish-ai-engineering-governance-guide/spec.md`
- 任务入口：`.trae/specs/establish-ai-engineering-governance-guide/tasks.md`
- 验收入口：`.trae/specs/establish-ai-engineering-governance-guide/checklist.md`

#### 3.4.2 闭环步骤
1. 规格对齐：在规格中确认治理主文档、分层边界、门禁与演进机制等需求。
2. 任务拆解：按 Task 1 ~ Task 5 分阶段推进，并维护依赖关系与子任务状态。
3. 产物实现：在 `docs/architecture/ai-engineering-governance.md` 完成双视图、闭环模板、门禁、度量与演进章节。
4. 清单验收：逐项核对 `checklist.md`，仅对具备可追溯证据的项进行勾选。
5. 状态收敛：将 Task 5 与 SubTask 5.1/5.2/5.3 置为完成，形成“规格 → 任务 → 文档 → 验收”的闭环。

## 4. 统一实施规范与架构约束

### 4.1 分层决策边界与允许变化范围
| 决策层级 | 决策对象 | 硬性约束 | 允许变化范围 | 典型产物 | 审批角色 |
| --- | --- | --- | --- | --- | --- |
| 架构层 | 领域边界、跨模块依赖、核心数据流、协议方向 | 不得绕过既有分层；不得引入未评估核心依赖；不得破坏兼容承诺 | 可调整模块交互模式与内部编排，但必须保留外部契约与回滚路径 | ADR/架构决策记录、边界图、迁移策略 | 架构师 + 产品工程负责人 |
| 模块层 | 模块目录组织、接口契约、状态机、错误域划分 | 必须遵循统一目录职责；接口输入输出与错误语义可测试；禁止跨模块读写内部状态 | 可替换模块内部实现技术；可新增模块内子目录，但职责必须单一 | 设计说明、接口定义、状态流转图 | 模块负责人 + 评审者 |
| 实现层 | 代码结构、命名、日志埋点、测试实现、脚本细节 | 必须遵循命名/日志/测试规范；禁止跳过错误处理与异常路径验证 | 可按语言特性选择具体模式；可在不改变契约前提下重构 | 代码变更、测试报告、验证记录 | 开发/AI 协作操作者 + 评审者 |

### 4.2 分层决策执行流程
1. 在任务设计阶段标注每项变更的决策层级（架构层/模块层/实现层）。
2. 为每项变更附加“硬性约束满足证据”和“可变项选择理由”。
3. 当变更同时覆盖多个层级时，按“架构层优先、模块层收敛、实现层落地”顺序评审。
4. 出现边界冲突时，以高层约束覆盖低层实现，不得以实现便利性反向修改高层边界。

### 4.3 统一实现规范
#### 4.3.1 命名规范
- 目录、模块、接口名称必须语义化，禁止使用无业务意义缩写。
- API、事件、配置项命名必须在同一领域内保持前后一致。
- 错误码与日志事件名必须稳定，禁止在无迁移策略下直接重命名。

#### 4.3.2 目录与职责规范
- 按“接口层 → 业务层 → 数据访问层/基础设施层”组织目录，禁止职责混放。
- 公共能力下沉到共享模块，业务特化逻辑保留在业务模块，禁止双向复制。
- 新增目录必须声明职责边界、依赖方向与归属模块。

#### 4.3.3 依赖与边界规范
- 依赖方向必须单向向下，不得出现循环依赖或跨层反向依赖。
- 引入第三方依赖需记录用途、替代方案、版本策略与风险评估。
- 跨模块调用优先通过稳定契约，禁止直接耦合对方内部实现细节。

#### 4.3.4 错误处理规范
- 错误必须可分类（参数、权限、资源、外部依赖、系统异常）并可追踪。
- 对外错误语义稳定且可文档化，内部错误细节不得泄露敏感信息。
- 异常路径与降级路径必须定义触发条件、返回语义与恢复策略。

#### 4.3.5 日志与可观测性规范
- 日志需包含上下文标识（请求标识、模块、关键资源标识）以支持问题定位。
- 日志级别使用一致口径，禁止将预期分支记录为错误级别噪声。
- 关键业务流程必须具备可观测点（日志、指标或追踪）并对应验收口径。

#### 4.3.6 测试与质量规范
- 每项功能至少覆盖主路径、异常路径与边界输入，不得仅验证成功路径。
- 接口变更必须补充契约一致性验证，回归风险点需有对应测试证据。
- 测试结果需可复现，包含执行入口、环境说明与失败处理记录。

#### 4.3.7 文档与交付规范
- 规格、实现、测试、发布信息必须在同一任务上下文可追踪。
- 重大行为变更需同步更新相关 README、设计文档与运维说明。
- 交付说明需明确变更影响面、兼容性结论与回滚方案。

## 5. 常见反模式与替代建议
| 反模式 | 风险 | 替代建议 |
| --- | --- | --- |
| 未标注决策层级直接实现 | 评审标准不一致，返工率上升 | 在任务开始时声明层级并绑定对应约束清单 |
| 接口层直连数据访问层 | 业务规则分散，难以复用与测试 | 通过业务层统一编排并集中处理规则 |
| 为赶进度绕过错误分类 | 故障定位慢，告警噪声高 | 建立统一错误域并保持外部语义稳定 |
| 新需求直接复制已有模块代码 | 技术债扩散，后续修复成本高 | 抽象公共能力并通过契约复用 |
| 跨模块读取内部字段或私有结构 | 模块耦合加深，升级易破坏 | 只通过公开契约/接口交换信息 |
| 仅补主流程测试，不测失败场景 | 线上异常缺乏保护，回归风险高 | 补齐异常路径与边界测试并纳入门禁 |
| 日志缺少上下文标识 | 问题排查链路断裂 | 统一日志最小字段集并按流程打点 |
| 文档滞后于实现 | 团队认知分裂，交接成本高 | 合并前同步文档并在交付证据中关联 |

### 5.1 反模式识别触发条件
- 评审中出现“无法说明层级归属”的变更。
- 变更说明中缺失错误语义、验证证据或回滚策略。
- 同类问题在两个迭代内重复发生且无规则修订记录。

### 5.2 替代方案落地要求
- 每个被识别反模式必须给出可执行替代方案、负责人和完成时限。
- 替代方案需补充验证证据，并在后续复盘中检查复发情况。
- 无法立即替代时，必须登记例外并附带过渡计划。

## 6. 合并前治理清单与门禁机制

### 6.1 合并前必检清单
| 检查维度 | 必检项 | 验证证据 | 阻断等级 |
| --- | --- | --- | --- |
| 架构一致性 | 变更已标注决策层级并符合 4.1 约束 | 任务模板中的层级标注、约束映射 | P0 |
| 架构一致性 | 跨模块依赖已评估且无反向/循环依赖 | 依赖变更说明、设计评审记录 | P0 |
| 实现一致性 | 接口契约与错误语义与规格一致 | 接口测试结果、契约比对记录 | P0 |
| 实现一致性 | 异常路径与降级路径已定义并可执行 | 用例截图、失败路径测试结果 | P1 |
| 规范一致性 | 命名、目录、日志、错误处理符合 4.3 规范 | 代码评审结论、抽样检查记录 | P1 |
| 规范一致性 | 文档、测试、发布说明已同步更新 | 变更说明、README/设计文档链接 | P1 |
| 交付完整性 | 已附最小闭环模板与交付证据 | 模板实例、PR 描述、验证附件 | P0 |

### 6.2 阻断条件
- 触发任一 P0 问题时必须阻断合并，禁止以“后续补齐”为由直接放行。
- 任一 P1 问题超过 2 项且未形成明确修复计划时阻断合并。
- 缺失回滚路径、缺失验证入口、缺失负责人任一项时阻断合并。
- 出现与既有硬性约束冲突且未完成升级决策时阻断合并。
- 阻断结论需在评审记录中标注问题编号、责任人、修复时限。

### 6.3 例外申请与审批流程
1. 申请人提交例外单，至少包含：例外条款、触发原因、影响面、风险等级、过渡方案、最晚补齐时间。
2. 模块负责人初审例外必要性与可行性；不满足“临时且可回收”原则直接驳回。
3. 架构师与产品工程负责人联合审批；涉及线上风险时需质量角色会签。
4. 通过后生成例外编号并写入任务与 PR；未通过则按阻断条件回退整改。
5. 例外到期前必须复核关闭；逾期未关闭自动升级为阻断项。

例外审批约束：
- 单个任务最多允许 2 个有效例外，且不得覆盖同一条 P0 约束。
- 例外有效期默认不超过 1 个迭代。
- 所有例外必须保留审批记录、风险评估与关闭结论。

### 6.4 验证方式
| 验证类型 | 最低要求 | 产出物 | 责任角色 |
| --- | --- | --- | --- |
| 静态检查 | 至少完成 lint 与类型/编译检查 | 命令与结果摘要 | 开发/AI 协作操作者 |
| 单元验证 | 关键主路径与边界输入可复现 | 测试用例与结果 | 开发/AI 协作操作者 |
| 集成验证 | 关键跨模块链路至少 1 条成功与 1 条失败路径 | 集成测试记录 | 测试/质量角色 |
| 手工验收 | 业务主流程、异常流程、回滚演练 | 验收记录与截图 | 产品工程负责人 |
| 发布验证 | 发布步骤、开关策略、回滚触发条件可执行 | 发布清单与演练结论 | 模块负责人 |

验证记录要求：
- 每类验证必须注明执行入口、执行环境、执行人、执行时间。
- 失败项必须附处置结论（修复、降级、登记例外）与复验结果。
- 验证证据需可追溯到具体任务与代码版本。

### 6.5 审查角色分工
| 审查活动 | 开发/AI 协作操作者 | 模块负责人 | 架构师 | 产品工程负责人 | 测试/质量角色 | 评审者 |
| --- | --- | --- | --- | --- | --- | --- |
| 准备最小闭环模板与证据 | R | A | C | C | C | I |
| 架构一致性审查与门禁裁决 | C | C | A | A | I | R |
| 实现与规范一致性审查 | R | A | C | C | C | A |
| 验证执行与缺陷确认 | R | C | I | C | A | C |
| 例外审批与到期复核 | C | R | A | A | C | I |
| 合并前最终放行 | I | A | C | C | C | R |

角色说明：
- R：执行与提交证据的直接责任角色。
- A：对结论负责并拥有最终签署权。
- C：提供专业评审意见并参与决策。
- I：同步知会，跟踪结果与风险状态。

### 6.6 可执行命令基线
为减少“需要执行但入口不明确”的歧义，治理校验默认使用以下仓库已有入口：

## 7. 演进与度量机制

### 7.1 治理指标定义与口径
| 指标 | 定义 | 计算口径 | 统计窗口 | 目标值 |
| --- | --- | --- | --- | --- |
| 偏差率 | 实际实现与规格/约束不一致的任务占比 | 偏差任务数 / 已完成任务数 | 每两周 | ≤ 10% |
| 返工率 | 已合并后因治理问题二次修改的任务占比 | 返工任务数 / 已完成任务数 | 每两周 | ≤ 15% |
| 缺陷率 | 与治理规则直接相关的线上或联调缺陷密度 | 治理相关缺陷数 / 已完成任务数 | 每两周 | ≤ 0.2 |
| 规范通过率 | 合并前一次性通过治理清单校验的任务占比 | 一次通过任务数 / 已校验任务数 | 每两周 | ≥ 90% |

指标采集要求：
- 每个任务在合并时必须登记是否触发偏差、返工、缺陷与一次通过状态。
- 指标统计口径必须固定，不得在同一统计窗口内临时调整定义。
- 指标结果需按模块维度与全局维度同时输出，保留可追溯的原始记录链接。

### 7.2 周期性复盘与规则更新流程
1. 每两周固定开展一次治理复盘，输入包含指标看板、阻断记录、例外清单、复发反模式清单。
2. 复盘时按“指标异常 → 根因分析 → 规则修订候选”形成问题闭环，明确责任人和截止时间。
3. 当任一核心指标连续两个周期未达标时，必须触发规则升级评审，不得仅登记风险后搁置。
4. 规则更新需经过“提案评审 → 小范围试运行 → 全量生效”三阶段，避免一次性全局切换风险。
5. 每次复盘必须输出结论纪要，至少包含：本周期指标结论、例外处理状态、规则调整清单、下周期跟踪项。

### 7.3 变更追踪与版本化机制
版本策略：
- 治理文档采用语义化版本：`主版本.次版本.修订版本`。
- 主版本用于破坏性规则变更；次版本用于新增规则或流程扩展；修订版本用于措辞澄清与非语义修正。
- 每次版本变更必须同步记录变更类型、影响范围、生效日期与回滚策略。

变更追踪要求：
- 每次规则调整必须生成唯一变更编号，格式：`GOV-YYYYMMDD-序号`。
- 变更记录必须关联触发来源（复盘编号/问题编号/例外编号）与对应证据链接。
- 变更生效后一个周期内必须完成效果复核，结论写入同一变更记录。
