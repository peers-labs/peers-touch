# 三端节点职责与协同模型

## 1. 文档目的
定义 Desktop、Mobile、Server 三类节点的职责边界、协作拓扑和故障降级策略。

## 2. 节点角色定义
## 2.1 Desktop 节点
- 定位：强节点、长期在线优先、可承担重计算与本地缓存主副本。
- 职责：社交主交互、Agent 重任务执行、Applet 管理与调试、历史数据维护。
- 能力：更高资源上限、更稳定网络、更强本地索引能力。

## 2.2 Mobile 节点
- 定位：轻节点、实时入口、间歇在线。
- 职责：即时社交交互、轻量 Agent 任务、消息收发与通知触达。
- 能力：低功耗优先、弱网容错、离线缓存与快速恢复。

## 2.3 Server 节点
- 定位：协调节点，不作为唯一数据真相。
- 职责：中继路由、全局索引、策略下发、审核审计、分发控制。
- 能力：高可用协同、统一治理、跨节点发现与同步辅助。

## 3. 协同拓扑
- 默认模式：Desktop/Mobile 与 Server 保持控制面连接，数据面可点对点。
- 弱网模式：端侧优先本地提交，后续通过同步协议收敛。
- 受限模式：当 P2P 不可用时，Server 中继作为兜底链路。

## 4. 关键链路
## 4.1 身份链路
- 端侧生成或托管密钥对。
- 登录后获取可轮换会话证明。
- 多设备以授权关系形成同一用户身份簇。

## 4.2 社交链路
- 内容生产在端侧先入本地日志。
- 异步同步至对端或 Server 中继。
- 通过冲突收敛规则统一状态视图。

## 4.3 Agent 链路
- 调度器根据任务类型选择执行位置（端/云）。
- 工具调用通过统一桥接协议执行。
- 结果、日志、错误统一归档到审计流。

## 5. 降级与容灾策略
- Server 不可用：维持本地功能 + 点对点能力，延迟执行全局能力。
- Mobile 离线：保留本地操作队列，恢复后自动重放。
- Desktop 离线：移动端接管轻任务，重任务转交 Server 执行。
- 节点冲突：按版本向量与业务优先级策略收敛。

## 6. 约束与边界
- Server 不直接承诺“单点最终真相”，只承诺收敛和可审计。
- 端侧必须支持最小离线能力集（身份、会话、缓存、重放）。
- 所有跨节点调用都必须可重试、可幂等、可观测。

## 7. 待决策问题
- 节点优先级策略：任务默认端侧执行还是云侧执行。
- P2P 拓扑形态：全网发现还是以信任图为基础的局部发现。
- 端侧存储配额策略：不同端的缓存淘汰规则是否统一。
